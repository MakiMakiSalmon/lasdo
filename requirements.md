# lasdo 要件定義 v1.0

> 作成: 2026-06-19 / 確定: 2026-06-20 / ステータス: MVP着手可（コア要件確定）

---

## 1. コンセプト（一言で）

**「予定」ではなく「実績」を記録し、自分の活動リズムをグラフで把握するツール。**
カレンダーが未来の計画を埋めていくものなら、lasdo は過去の実際の時間の使い方をタイマーで埋めていく。
そのデータを **「いつ・どれだけアクティブだったか」が一目で分かるグラフに可視化する** のが主目的。

キーワード: _カレンダーの逆引き_ / 実績ベース / **活動リズムの可視化**

> 補足: 「サボり時間」は `24h − アクティブ時間` で自動的に出るおまけにすぎず、主役ではない。
> 力点は「サボりを責める」ことではなく「**自分のアクティブ/非アクティブのパターンを見える化する**」ことにある。

---

## 2. 解決したい課題（動機）

- 「今日は忙しかった」という体感と、実際の作業時間が乖離している。
- 予定（カレンダー）は立てても、実績がどうだったかは記録されず振り返れない。
- 自分の活動リズム（いつ・どれだけ働けているか）が見えないので、改善のしようがない。

→ **実績を計測・可視化することで、体感と現実のギャップを埋める。**

---

## 3. ターゲットユーザー

- 第一ユーザーは自分（個人の自己管理ツールとして開始）。
- 想定: 集中して働きたいが、実態がつかめずモヤモヤしている人。
- （将来）フリーランス・リモートワーカーなど、自己申告で時間を管理する人。

---

## 4. コア機能

### 4.1 ハイブリッド記録
- **リアルタイム計測**: 作業開始時にタイマーを開始／停止する（ストップウォッチ的）。
- **後から補完**: 押し忘れた区間を、後から「何時〜何時」と手動で埋められる。
- 両者を同じタイムラインに統合して扱う。

### 4.2 記録の単位：アクティブ区間のみ（種別なし）
- 記録するのは **「アクティブだった区間」だけ**。ブロックに種別（作業/非作業など）は持たせない。
- **タイマーが回っている＝アクティブ／記録がない＝非アクティブ**、というシンプルな二分。
- 睡眠・昼食・休憩は「記録しない」だけ。わざわざ非アクティブとして入力する必要はない。
- → データは `開始 + 終了` の最小2項目で成立する（種別カラム不要）。

### 4.3 状態モデル：2状態（記録あり / 記録なし）
| 状態 | 意味 | 見え方（例） |
|------|------|--------------|
| アクティブ | 記録された区間（タイマー or 手動入力） | 濃い色の帯 |
| 非アクティブ | 記録がない時間（= 記録の不在として算出） | 無色/空白 |

- 「サボり時間」= `24h − アクティブ時間` で自動的に出るおまけ。主要指標にはしない。

### 4.4 設計原則：システムは時間を自動分類・自動挿入しない ★重要
- システムが「夜だから睡眠だろう」等と**勝手に非アクティブ区間を作らない**。
  （例: 徹夜で作業している人を自動でサボり扱いするとUXが壊れる）
- 保存するのは **ユーザーが記録した区間だけ**。非アクティブはあくまで「記録の不在」として表示時に算出する。
- 深夜をまたぐアクティブ区間（例 23:00〜翌2:00）も、**分割せず1ブロックのまま保存**。
  日付またぎは「表示・集計のときに切って描く」だけの処理で、データには手を入れない。

### 4.5 可視化（★本ツールの主目的）
実績データを以下のグラフで多面的に見せることがコアバリュー。

| ビュー | 内容 | 目的 |
|--------|------|------|
| **1日タイムライン** | カレンダーのように、その日のアクティブ/非アクティブ帯を表示（軸は5:00〜29:00固定／7章参照） | その日の過ごし方を直感的に把握 |
| **曜日別棒グラフ** | 曜日ごとの合計アクティブ時間を比較 | 「何曜日に働けているか」を比較 |
| **箱ひげ図** | 各日のアクティブ開始/終了時刻の分布（中央値・四分位・最小最大）を曜日 or 週単位で | 活動リズムの傾向とブレを把握 |
| **ヒートマップ（任意）** | 時間帯 × 日 のマトリクスでアクティブ濃淡を表示 | 習慣・パターンを俯瞰 |

- これらは記録データ（時間ブロックの集合）から集計して描画する（集計方法は6.4）。
- 実装優先順: **①1日タイムライン → ②曜日別棒グラフ → ③箱ひげ図**。MVPで全部は作らない。

### 4.6 画面構成（情報設計）
グラフを全部対等に並べると、毎日使う「記録」のときに邪魔になる。見る頻度で **2画面に分離** する。

| 画面 | 置くもの | 使用頻度 |
|------|----------|----------|
| **記録画面（メイン）** | タイマー（4.7）＋ **今日の1日タイムライン** だけ | 毎日・常時 |
| **分析画面（振り返り）** | 曜日別棒グラフ・箱ひげ図など分析系グラフ | 週1〜たまに |

- 記録画面はミニマルに保つ（タイマーと今日の帯が見えれば十分）。**ブロック一覧（リスト）は置かない**（常駐画面のノイズ回避）。
- 編集・履歴は「記録を編集 →」から**別画面**に逃がす。
- 分析系グラフは普段は隠れていて、分析画面を開いたときだけ見る。
- 分析画面内の期間切替（週/月/全期間）や、箱ひげ図を「全体1本 vs 曜日別」にするかは → 9章で詰める。

### 4.7 記録画面のタイマーUI：がんばりゲージ式の円形タイマー
コアの操作系。ゼルダBotWのスタミナ（がんばり）ゲージを参照した円形タイマー。

- **中心 = 開始/停止ボタン兼・現在の作業時間表示**（タップでトグル、中央に経過時間）。
- **内側の太いリング = 現在の単位時間の進捗**（単位を満たすと1周）。
- **単位時間（＝1セッションの長さ基準・固定値）= 25分に確定**（定数化し後から変更可）。1つ達成するごとに外側に細い同心円リングが1本増える。
  - 補足: 25分は「リングが1周する基準」にすぎず、休憩を強制する仕掛けではない（lasdo は休憩を記録しない）。
  → リングが育つほど「がんばった」が見える。**超過＝良いこと／応援のイメージ**（色は緑系）。
- **リングが増えすぎる前に `×N` 表記に集約**（数時間〜数十時間でも破綻しないように）。
  - 集約後は**外周リングを1本だけに戻し**、達成回数は `×N` で表す（リングを何本も残さない＝視覚ノイズ回避）。
  - **しきい値 = 外周3本まで描画し、4本目以降は `×N` に集約**（25分単位なら約100分で集約）。
- 「単位時間」の定義は **(A) 1セッションの長さ基準** に確定（時計の1時間枠ごとの密度=(B)はタイムラインで分かるので採らない）。

### 4.8 プラットフォーム別UIの方針
- **スマホとPCでUIは根本的に作り分ける**前提（同一レイアウトの流用はしない）。
  - 例: スマホ＝円形タイマー中心の片手操作／PC＝より情報密度を上げてよい。
- データ・ロジックは共通、画面（プレゼンテーション層）を出し分ける。

## 5. コア体験のユースケース（ストーリー）

> 第一ユーザー（本人）の平日1日を例に、記録〜振り返りの流れを通しで書く。

### 朝〜日中：記録する
1. **8:50 起床後、PCのブラウザでlasdoを開く。** 睡眠は記録しない（記録がない時間＝自動的に非アクティブ）。
2. **9:10 仕事開始。タイマーの中心を押して開始。** （アクティブ区間の記録が始まる）
3. **10:30 集中が切れてコーヒー休憩。中心を押して停止。** 9:10〜10:30がアクティブとして記録される。
4. **10:45 作業再開、また開始。** ……これを1日くり返す。
5. **13:00 昼食。停止し忘れたまま食べ始めてしまった。**
   → 後で気づいて、誤って延びたブロックの**終了時刻を12:55に修正**（or 昼食ぶんを削る）。編集は別画面（4.6）から。
6. 押し忘れ・押し間違いは日中いつでも、ブロックを編集（追加/時刻変更/削除）して直せる。

### 夜：振り返る
7. **22:00 1日を振り返る。** 「1日タイムライン」で今日のアクティブ帯を見て「午後スカスカだった」と気づく。
8. **週次で見る。** 「曜日別棒グラフ」で「水曜が極端に低い」と気づく。
9. **傾向を見る。** 「箱ひげ図」で「開始時刻のばらつきが大きい＝リズムが不安定」と把握。

### このストーリーから逆算される要件
- **記録操作**: ①開始/停止（円形タイマーの中心）②ブロックの追加/編集/削除（別画面での手動補完・修正）。
- **データに必要な項目**: 開始時刻・終了時刻のみ（→ 4.2 / 6.1）。
- **編集の自由度**: 過去のブロックをいつでも直せること（押し忘れ・押し間違い前提）。
- **日付またぎ**: 深夜作業のブロックはそのまま保存し、表示時のみ切る（→ 4.4 / 6.3）。

## 6. データモデル（最小スキーマ）

### 6.1 エンティティ
記録は「アクティブだった時間区間」の集合。これだけ。

```
TimeBlock {
  id:    識別子
  start: 開始日時（タイムゾーン込み）
  end:   終了日時（start より後）
}
```
- 種別カラムは持たない（→ 4.2）。非アクティブはブロックの「不在」で表す。
- 1区間 = 1つの連続したアクティブ時間。

### 6.2 区間ルール
- **重なり（オーバーラップ）は禁止**。新規追加・編集で既存区間と重なったら **自動マージ**して1つの連続区間にする。
  - 例: 9:00〜11:00 がある状態で 10:30〜12:00 を足す → 9:00〜12:00 に統合。
  - 隣接（10:00〜11:00 と 11:00〜12:00）も任意でマージしてよい。
- これにより「アクティブ合計 = 各区間長の単純な総和」で求まり、二重計上が起きない。
- `end > start` を常に保証（ゼロ幅・逆転は不可）。

### 6.3 日付またぎ・1日の区切り
- 深夜をまたぐ区間（例 23:00〜翌2:00）も **分割せず1ブロックとして保存**（→ 4.4）。
- **1日の区切り = 朝5:00起点**（5:00〜翌4:59を「1日」とする）。夜型・徹夜でも崩れにくい。
- 「1日タイムライン」「曜日別集計」で日付境界が要るときだけ、**表示・集計の段階で**この基準で区間を切る。

### 6.4 集計の考え方（グラフの素）
- **アクティブ合計（日/週）**: 対象期間に重なる区間長を合算。
- **曜日別**: 各区間を日付（5:00起点）で割り当て（またぎは分割）、曜日ごとに合算。
- **開始/終了時刻の分布（箱ひげ図）**: 各日の「最初の区間の開始」「最後の区間の終了」を日ごとに集め、その分布を取る。
  - 「開始」の箱 = 各日のアクティブ開始時刻の分布／「終了」の箱 = 各日のアクティブ終了時刻の分布。
  - 既知の穴: 朝/深夜の極短区間があると開始/終了が実態とズレる。当面は素直な定義のまま運用し、困れば「短区間を無視」「一定以上の空白で区切る」等の補正を後で足す（→ 9章）。

## 7. アーキテクチャ方針

**「軽量キャプチャ + 集約データ + リッチな分析」の分離構成。**
プラットフォームを1つに絞らず、データを中心に据えて段階的に拡張する。

```
        ┌─────────────┐
        │  集約データ  │  ← どのクライアントから入れても同じタイムラインに乗る
        └─────────────┘
          ↑    ↑     ↑
   ┌──────┘    │     └────────────┐
[Webコア]   [スマホ]        [PC常駐エージェント]
タイマー＋   Webコアを      （任意・後付け）
手動補完＋   軽くタイマー    アクティブウィンドウ/
分析・確認   として使用       アプリ利用状況から
（PC/スマホ  確認もブラウザ   アクティブ/非アクティブを
共通）                      自動判定 → データに流し込む
```

### 7.1 フェーズ分け
- **フェーズ1（MVP）**: Webコア。タイマー・手動補完・1日タイムライン・日/週集計。
  ブラウザで動くのでスマホからも同じものを使える（UIはプラットフォーム別に出し分け／4.8）。
- **フェーズ2（任意拡張）**: PC用デスクトップ常駐エージェント。
  画面/アプリの利用状況からアクティブ/非アクティブを自動入力。価値が見えてから着手。
- スマホ専用ネイティブアプリは当面作らない（Webで代替、軽さを保つ）。

### 7.2 この構成の利点
- 今すぐプラットフォームを確定しなくてよい。
- 重い自動判定機能（PC）と軽いタイマー（スマホ）を、用途ごとに分離できる。

## 8. やらないこと（スコープ外・現時点）

- 他人との共有・チーム機能（まずは個人ツール）。
- 予定（未来の計画）の管理 ← 既存カレンダーの領分。lasdo は実績特化。
- 記録ブロックの種別/カテゴリ/タグ分類（アクティブ区間のみ・種別なし。将来拡張の余地は残す）。
- スマホネイティブアプリ（Webで代替）。

## 10. 技術スタック（確定 2026-06-21）

| 領域 | 採用 | 補足 |
|------|------|------|
| フロントエンド | **React + Vite + TypeScript** | PC/スマホは画面を出し分け（4.8） |
| データ保存 | **IndexedDB（Dexie）** | **Repository 層の裏に隠す**。UI・ロジックは抽象インターフェース越しにのみ触る |
| 同期（将来） | **Supabase**（後付け） | スマホ同期/PCエージェント（フェーズ2）が必要になったら Repository 実装を差し替え |
| 円形タイマー・1日タイムライン | **自前 SVG** | 既製グラフに合わないbespoke UI（4.7） |
| 棒/箱ひげ/ヒートマップ | **ECharts**（`echarts-for-react`） | 箱ひげ・ヒートマップを標準装備 |
| 日付・時刻計算 | **date-fns** | 5:00起点・日付またぎ・曜日集計（6.3/6.4） |
| 状態管理 | **Zustand** | タイマー/ブロックの薄いストア |
| テスト | **Vitest** | 区間マージ・集計ロジックの単体テスト |
| スタイル | **CSS Modules** | 当面は素のCSS（カスタムデザイン主体） |
| パッケージマネージャ | **npm** | — |

> 保存方針の要点: 要件7の「集約データ」構成を見据えつつ、MVPは単一端末で最速に作る。**`TimeBlockRepository`（list/add/update/delete）を境界**にし、最初は IndexedDB 実装、後で Supabase 実装に差し替えても UI/集計は無傷、という設計で「速く始める」と「将来詰まない」を両立する。
