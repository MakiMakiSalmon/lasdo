# CLAUDE.md

このファイルは Claude Code がこのリポジトリで作業する際の前提を共有するためのもの。

## プロジェクト概要

**lasdo** = 「予定」ではなく **実績** を記録し、自分の **活動リズムをグラフで可視化** するツール。
カレンダーの逆引き：タイマーで過去の実際の時間の使い方を埋めていく。

詳細な要件は [requirements.md](requirements.md)（v1.0 確定）が正典。迷ったらそちらを参照する。

## 設計上の力点（誤解しやすいので明記）

- **主目的は「活動リズムの可視化」**。1日タイムライン／活動カレンダー（草＋曜日平均）／時間帯ヒートマップが主役。
- 「**サボり時間**」は `24h − アクティブ時間` で出る **おまけ**。主指標にしない・責める設計にしない。

## 絶対に守る設計原則

- **システムは時間を自動分類・自動挿入しない**。「夜だから睡眠だろう」等で勝手に非アクティブ区間を作らない（徹夜作業を自動でサボり扱いするとUXが壊れる）。
- **保存するのはユーザーが記録したアクティブ区間だけ**。非アクティブは「記録の不在」として表示・集計時に算出する。
- **データは `開始 + 終了` の2項目のみ**。種別/カテゴリ/タグのカラムは持たない。
- **深夜またぎ区間（例 23:00〜翌2:00）は分割せず1ブロックで保存**。日付境界は表示・集計時にだけ切る（データには手を入れない）。
- **区間の重なりは禁止 → 自動マージ**。`end > start` を常に保証。これで「アクティブ合計 = 各区間長の総和」が成り立つ（二重計上しない）。
- **1日の区切り = 朝5:00起点**（5:00〜翌4:59）。タイムライン軸は 5:00〜29:00 固定（日ごと比較のためズームしない）。

## 画面構成

- **記録画面（毎日・ミニマル）**: 円形タイマー（ゼルダBotWのがんばりゲージ風）＋ 今日のタイムラインだけ。ブロック一覧リストは置かない。
- **分析画面（たまに）**: 活動カレンダー（草＋曜日ごとの平均/日）・今週・時間帯ヒートマップ。
- スマホ/PC は UI を根本的に作り分ける（データ・ロジックは共通、画面を出し分け）。

## アーキテクチャ

- 「軽量キャプチャ + 集約データ + リッチな分析」の分離構成。
- **フェーズ1（MVP）**: Webコア（タイマー・手動補完・1日タイムライン・日/週集計）。実装優先順は ①1日タイムライン → ②活動カレンダー（草・曜日平均）→ ③時間帯ヒートマップ（箱ひげ図は保留）。
- **フェーズ2（任意）**: PC常駐エージェントで自動判定。価値が見えてから。
## 技術スタック（確定・requirements.md 10章）

- **React + Vite + TypeScript** / 状態管理は **Zustand** / テストは **Vitest** / スタイルは **CSS Modules** / pkg は **npm**。
- 保存は **IndexedDB（Dexie）を `TimeBlockRepository`（list/add/update/delete）の裏に隠す**。UI・集計ロジックはこの境界越しにのみデータへ触る。将来 **Supabase** 実装に差し替えてスマホ同期/PCエージェントに対応する想定。
- 円形タイマー・1日タイムライン・分析チャート（活動カレンダー／今週／ヒートマップ）はすべて **自前描画**（SVG／CSS）。ECharts は使わない（依存も削除済み）。
- 日付・時刻（5:00起点・日付またぎ・曜日集計）は **date-fns**。

## 開発フロー

- **PR ベース運用**。`main` への直接 push は不可（保護済み）。feature ブランチ → PR → セルフマージ。
- コミット/PR は Conventional Commits（`feat:`／`fix:`／`chore:`／`docs:`／`refactor:`）。要約は日本語で簡潔に。
- 詳細は [docs/git-workflow.md](docs/git-workflow.md)。
