## 2. 設計の前提（要件からの不変条件）

実装中ずっと守る原則。詳細は requirements.md 4.4 / 6.2 / 6.3。

1. **自動分類・自動挿入をしない**。保存するのはユーザーが記録したアクティブ区間だけ。
2. **データは `start + end` の2項目のみ**。種別/カテゴリ/タグは持たない。
3. **深夜またぎは1ブロックで保存**。日付境界は表示・集計時だけ切る（データは無加工）。
4. **区間の重なり禁止 → 自動マージ**、`end > start` を常に保証。
5. **1日の区切り = 朝5:00起点**。タイムライン軸は 5:00〜29:00 固定。
6. **可視化が主目的**、「サボり時間」はおまけ（責める設計にしない）。

## 3. 全体アーキテクチャ

requirements.md 7章「軽量キャプチャ + 集約データ + リッチな分析」の分離構成を、MVPでは**単一端末・レイヤード構成**として実装する。

```
┌──────────────────────────────────────────────┐
│  Presentation 層（画面・自前 SVG/CSS 描画）      │  PC/スマホで出し分け（4.8）
│   記録画面 / 分析画面 / 編集画面               │
└───────────────┬──────────────────────────────┘
                │ 参照・操作
┌───────────────▼──────────────────────────────┐
│  State 層（Zustand ストア）                     │  タイマー状態・ブロック集合
└───────────────┬──────────────────────────────┘
                │ ドメイン規則を適用して呼ぶ
┌───────────────▼──────────────────────────────┐
│  Domain 層（純粋関数）                          │  マージ・集計・日付境界処理
│   timeBlock / aggregation / dateBoundary       │  ← 副作用なし・テストの主戦場
└───────────────┬──────────────────────────────┘
                │ CRUD のみ
┌───────────────▼──────────────────────────────┐
│  Repository 層（TimeBlockRepository）           │  保存先を隠す境界
│   IndexedDB 実装（将来 Supabase に差し替え）    │
└──────────────────────────────────────────────┘
```

**依存方向は上から下の一方向**。Domain 層は他のどの層にも依存しない（純粋関数のみ）。

### 3.1 各層の責務

| 層 | 責務 | 依存してよい先 | 主な配置 |
|----|------|----------------|----------|
| Presentation | 画面描画・ユーザー操作の受付。プラットフォーム別に出し分け | State, Domain（表示用集計） | `src/ui/` |
| State | タイマー稼働状態・ブロック集合の保持。Domain 規則を適用して Repository を呼ぶ | Domain, Repository | `src/store/` |
| Domain | マージ・集計・日付境界の純粋計算。副作用ゼロ | （なし） | `src/domain/` |
| Repository | 永続化の抽象境界（CRUD のみ。ドメイン規則は持たない） | （実装が IndexedDB/Dexie） | `src/data/` |

> **重要**: マージ等のドメイン規則は Repository に置かない。Repository は素の CRUD に徹し、`mergeBlocks` は State 層が add/update の前に適用する（既存コードのコメント方針どおり）。

## 4. 画面構成（情報設計）

requirements.md 4.6 に従い、見る頻度で画面を分ける。

| 画面 | 置くもの | 頻度 | フェーズ1 |
|------|----------|------|-----------|
| **記録画面**（メイン） | 円形タイマー + 今日の1日タイムライン | 毎日・常時 | ✅ |
| **分析画面** | 活動カレンダー（草＋曜日平均）・今週・時間帯ヒートマップ | 週1〜 | ✅ |
| **編集画面** | ブロックの追加/時刻変更/削除 | たまに | ✅ |

- 記録画面はミニマルに保つ。**ブロック一覧リストは置かない**（4.6）。
- 編集・履歴は記録画面から別画面へ逃がす。
- PC/スマホで UI を作り分ける（データ・ロジックは共通／4.8）。

### 4.1 画面遷移

```
        ┌─────────────┐  「記録を編集 →」  ┌─────────────┐
        │  記録画面    │ ─────────────────▶ │  編集画面    │
        │ (タイマー+   │ ◀───────────────── │ (追加/修正/  │
        │  今日の帯)   │      戻る           │  削除)       │
        └──────┬──────┘                     └─────────────┘
               │ 「分析 →」
               ▼
        ┌─────────────┐
        │  分析画面    │
        │ (草/ヒート)  │
        └─────────────┘
```

- **遷移の実装手段**: react-router は入れず、**Zustand の `view` 状態（'record'|'analysis'|'edit'）で出し分け**る。3画面のMVPには最軽量。URL同期が要れば後で router に移行可（詳細設計 5.3）。

## 5. 主要コンポーネント（Presentation）

> **プラットフォーム方針（MVP）**: requirements.md 4.8 は「PC/スマホで根本的に作り分け」だが、**MVPは PC 先行**（スマホは後回し）。データ/ロジック/ストアは共通なので、後からスマホ用 Presentation を足して分離できる。

実装優先順は requirements.md 4.5: **①1日タイムライン → ②活動カレンダー（草・曜日平均）→ ③時間帯ヒートマップ**（箱ひげ図は保留）。

| コンポーネント | 種別 | 技術 | 優先 |
|----------------|------|------|------|
| 円形タイマー（がんばりゲージ） | 自前 | SVG | 高 |
| 1日タイムライン（5:00〜29:00） | 自前 | SVG | ① |
| 活動カレンダー（草＋曜日平均） | 自前 | SVG/CSS | ② |
| 時間帯ヒートマップ | 自前 | CSS | ③ |
| 今週チャート | 自前 | SVG | 中 |
| 箱ひげ図 | （保留） | — | 保留 |
| ブロック編集フォーム | 自前 | HTML | 中 |

- 円形タイマー・1日タイムラインは bespoke UI なので**自前 SVG**。
- 分析チャート（活動カレンダー/今週/ヒートマップ）も**自前描画**（SVG/CSS）。ECharts は不使用（依存削除済み）。

## 6. データフロー

### 6.1 記録（タイマー開始→停止）

```
[中心ボタン押下=開始] → State: 稼働開始時刻を保持（まだ永続化しない）
        … 経過 …
[中心ボタン押下=停止] → State: {start, end} を作る
        → Domain: 既存区間と mergeBlocks で重なり解消
        → Repository: add / update（マージ結果を反映）
        → State 更新 → タイムライン再描画
```

### 6.2 手動補完・編集

```
[編集画面で時刻入力] → Domain: isValidBlock で end>start 検証
        → Domain: mergeBlocks（重なれば統合）
        → Repository: add/update/delete → State 更新
```

### 6.3 可視化（読み取り）

```
Repository.list() → State: 全ブロック保持
        → Domain: 表示・集計時だけ日付境界(5:00起点)で切る／曜日集計／分布算出
        → Presentation: 自前描画（SVG / CSS）へ
```

データ本体は無加工。境界切り・集計は**読み取り側の純粋関数**でのみ行う。

## 7. データ保存方針

- MVPは **IndexedDB（Dexie）** 単一端末。
- `TimeBlockRepository`（list/add/update/delete）を境界にし、UI/集計はこの越しにのみ触る。
- 将来スマホ同期/PCエージェントが要れば **Supabase 実装に差し替え**（`src/data/repository.ts` の1行差し替えで移行、UI/集計は無傷）。
- 永続化は日時を **epoch ミリ秒**で保存（タイムゾーン非依存）。ドメインでは `Date` に変換。

## 8. ディレクトリ構成（方針）

```
src/
  domain/      純粋関数（timeBlock, aggregation, dateBoundary）+ テスト
  data/        Repository インターフェース + IndexedDB 実装
  store/       Zustand ストア（block / timer / view）   ← 新設
  ui/          画面・コンポーネント（MVPはPC版）  ← 新設
    timer/     円形タイマー(SVG)
    timeline/  1日タイムライン(SVG)
    analysis/  分析チャート（自前 SVG/CSS）
    edit/      編集フォーム
```

既存（`domain/` `data/`）はそのまま活かし、`store/` `ui/` を追加していく。
将来スマホ用を作るときは `ui/` 配下を platform 別に分ける（store/domain/data は共通のまま）。

## 10. 確定した設計判断（2026-06-21、分析は2026-06-30改定）

requirements.md 9章の残課題を消化。詳細は [detailed-design.md](detailed-design.md) 9章。

- タイマー単位時間 = **25分**、`×N` 集約 = 外周3本まで描画/4本目以降集約。
- 分析: 期間 = **固定窓（直近12週）**（プリセット廃止）、構成 = 活動カレンダー（草＋曜日平均/日）＋今週＋時間帯ヒートマップ。曜日別棒は欄外平均へ統合して廃止、箱ひげは保留。
- 箱ひげ極短区間補正（保留・実装時に適用）= 開始/終了判定で **5分未満の区間を無視**（duration不影響）。
- フェーズ1画面 = **記録／分析／編集の3つ**。
- タイマー稼働中の**リロード耐性をMVPに含める**（localStorage退避）。
- 名前の由来 = **"last do"**。
- 画面遷移 = **Zustand の `view` 状態で出し分け**（react-router 不使用）。
- プラットフォーム = **MVPは PC 先行**（スマホは後回し。store/domain/data は共通）。
- ブロック永続化 = **MVPは全置換方式（clear + bulkAdd）**。件数が増えたら差分 reconcile に切替（詳細設計 5.1）。

残課題（任意・将来）: 箱ひげ図の実装、任意期間レンジ、スマホ版UI、差分reconcileへの移行。
